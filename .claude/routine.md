# Routine — project-specific operations

This file documents the project-specific commands, conventions, and quirks for **WeatherApp_TOP**. It supplements `routine-base.md` (which has the universal discipline) with project-specific details Claude runs need to know.

If `routine-base.md` is "how Claude runs work in general," this file is "how Claude runs work for *this* repo specifically."

---

## Project identity

**Name:** WeatherApp_TOP

**Description:** A three-day weather forecast page. Pulls a 3-day forecast from WeatherAPI.com (defaulting to zip 98052), shows one day at a time — condition icon, high/low, feels-like, humidity, rain chance, wind — with left/right arrows to page between days and a search box to change location.

**Stack:** vanilla JS + Webpack 5. CommonJS, no framework, no router, no state library.

---

## Working directories

**Working directory:** `WeatherApp_main/`

All preflight and test-verification commands run from `WeatherApp_main/`. `package.json` lives there, not at the repo root — `npm install` and `npm test` fail with ENOENT anywhere else.

- **Source code:** `WeatherApp_main/src/`
- **Tests:** none — this project has no test suite.
- **Build output:** `WeatherApp_main/dist/` — gitignored, regenerated by every build. Never commit it.

When the user references a file by bare name (e.g. `logic.js`), it lives in `WeatherApp_main/src/`.

## Commands the routine runs

**Install dependencies (if needed before tests/build):**
```
npm install
```

**Run tests:**
```
npm test
```

There is no test suite. `npm test` is aliased to `npm run build` (webpack), so it is a compile check: it catches syntax errors, broken imports, and unresolvable assets, and nothing else. A green `npm test` means "it compiles," not "it works."

`<implementation>` step 5 in routine-base.md asks for a regression test before every `Type: bug` fix. There is nowhere to put one — no framework, no `tests/` directory. Skip that step, write "No test coverage exists for this project; manual review recommended" in the PR body's Testing section, and do not add a test framework unless an entry explicitly asks for it — `<hard_constraints>` forbids unannounced dependencies.

**Build:**
```
npm run build
```

**Deploy target:** GitHub Pages

The `deploy.yml` workflow handles deployment automatically on merges to `main`. Claude runs do not need to invoke deploy directly.

## Project-specific conventions

- **There is no HTML source file.** `html-webpack-plugin` generates `dist/index.html` from nothing but a title. Every element in the app is built in `index.js` with `document.createElement`, given an `id`, and appended. New UI means: create the element, assign its `id`, append it to the right parent in `index.js`, and add a matching `#id` rule in `style.css`.
- **Layout is CSS Grid keyed to element IDs.** `style.css` targets `#id` and assigns `grid-area`. Adding an element to a grid parent without adding it to that parent's `grid-template-areas` will silently place it in an implicit row.
- **The spacer divs are load-bearing.** `space1`, `spacer1`, `spaced1`–`spaced5`, `upperLeft`, `typeUpper`, `dateSpace1`, and friends are empty by design — they're named grid areas doing the spacing. Do not "clean up" empty divs.
- **`index.js` builds the DOM; `indexChanges.js` mutates it.** Every post-render update goes through `changeWeatherInfo(alldays)` in `indexChanges.js`, which re-queries by id and sets `textContent`. Don't write display updates into `index.js` or `logic.js`.
- **Data flow is one-directional:** `logic.js` fetches → constructs `Day` objects (`day.js`) → stores them on `alldays` as `day1`/`day2`/`day3` → hands `alldays` to `changeWeatherInfo`. Keep new data-shaping in `logic.js`, not in the render path.
- **No dependencies get added** without an entry that explicitly says so. `lodash` is imported in `index.js` and `indexChanges.js` and used in neither — don't take that as license.

## Known fragile areas

- **`#weatherLocation` displays the weather; `#weatherType` displays the location.** The IDs are exactly backwards from what they render, and `indexChanges.js` compensates by swapping the variable names on lookup (`weatherType = getElementById('weatherLocation')`). The net result is correct on screen. Anyone reading either file in isolation will get it inverted. Don't rename one side without the other.
- **The four webpack entry points duplicate shared modules.** `webpack.config.js` declares `index`, `changes`, `logic`, and `day` as separate entries, so `logic.js` and `indexChanges.js` are compiled into `index.bundle.js`, `changes.bundle.js`, and `logic.bundle.js` — and the generated `index.html` loads all four. `forecastLogic` is an IIFE that calls `initialAPICall()` at module-eval time, so a page load fires three identical WeatherAPI requests, and `changeWeatherInfo` registers the arrow and submit listeners three times with no one-shot guard. It appears to work because each closure carries its own `dayCounter` and they converge on the same value. Do not fix this incidentally — it needs its own entry (collapse to a single `index` entry point).
- **`getCurrentTime()` always returns 0.** It parses `date.toUTCString().substring(17,19)` from `date_epoch`, which is midnight, so `feelsLike` and `humidity` always read `json.hour[0]` — midnight's values, never the current hour. Known, unfixed, needs its own entry. Don't silently "correct" it as part of unrelated work.
- **`logic.js` and `indexChanges.js` import each other.** The cycle resolves today only because `indexChanges.js` uses `forecastLogic` inside a click handler rather than at module scope. Moving that call to module level will break the load.

## Manifest generator

This project publishes `src-manifest.json` to its deployed site so the in-app Claude assistant's file picker can discover its source files. The generator runs as part of `deploy.yml`.

**Variant:** `gen-src-manifest.js` (CommonJS — `package.json` has no `"type": "module"`)

**Location:** `WeatherApp_main/scripts/gen-src-manifest.js`. It must sit beside `package.json`, not at the repo root: the script derives its source and output directories from `path.resolve(__dirname, '..')`, so from anywhere else it scans a nonexistent `src/`, writes to the wrong `dist/`, and exits 0 while publishing an empty manifest. `deploy.yml` runs it as `node scripts/gen-src-manifest.js` with `working-directory: WeatherApp_main`.

The manifest step must run **after** the build — `webpack.config.js` sets `output.clean: true`, which wipes `dist/` and would delete a manifest generated first.

The manifest publishes bare filenames; the worker's `resolveAttachPath` prepends the configured `srcPrefix` to construct full repo-relative paths at fetch time.

## Notes for the in-app Claude assistant (Sonnet)

- Single page, no routing, no framework. Four files carry everything: `index.js` (builds the whole DOM), `indexChanges.js` (all post-render updates + the arrow/submit listeners), `logic.js` (the WeatherAPI fetch and the `alldays` store), `day.js` (a 10-field `Day` class).
- The UI is one CSS Grid tree from `#outerContainer` down, addressed entirely by element ID. Almost every visual question is answered in `style.css` plus the `createElement`/`appendChild` block in `index.js`.
- The three forecast days are all fetched up front in one call; the arrows page between already-loaded `alldays` entries and never re-fetch. Only the search box triggers a new request.
- Responsive support is two `@media` blocks at 768px and 480px at the bottom of `style.css`. Everything above them is fixed-width desktop layout.

## Notes for Opus (Claude Code in CI)

- Run `npm install && npm test` from `WeatherApp_main/` before opening the PR. `npm test` is a webpack build — treat "command not found" or a module-resolution error as an un-hydrated environment (install and retry), not a red suite.
- Never commit `WeatherApp_main/dist/`. It's gitignored, and `npm test` regenerates it on every run — a dirty `dist/` after testing is expected, not a change you made.
- Don't modify `webpack.config.js` or `package.json` without an entry that explicitly asks for a build or dependency change.
- Don't modify anything under `.github/workflows/`. The run's `GITHUB_TOKEN` cannot push changes there; the push will be rejected. If an entry seems to need a workflow edit, report it and exit rather than working around it.
- When an entry names a UI element, search `style.css` for its `#id` first — the grid rule usually tells you more about the element's role than the `createElement` line does.

## Deployment quirks

- **`publicPath` is hardcoded to the repo name.** `webpack.config.js` sets `publicPath: '/WeatherApp_TOP/'` to match the GitHub Pages URL. Renaming the repo breaks every asset path in the deployed build.
- **The deployed build is a development build.** `mode: "development"` with `devtool: 'inline-source-map'` means production ships unminified bundles with inline sourcemaps (`index.bundle.js` alone carries all of lodash). This is the current state, not an accident to tidy up — switching to production mode is a deliberate decision that needs its own entry.
- **The WeatherAPI key is hardcoded in `logic.js`** and therefore committed and shipped inside the bundle. Any entry touching that fetch should leave it alone unless the entry is specifically about moving it.
- `deploy.yml` publishes `./WeatherApp_main/dist` to the `gh-pages` branch; GitHub's own `pages-build-deployment` then serves it. A green Deploy plus a green pages-build-deployment is what "shipped" means here.
